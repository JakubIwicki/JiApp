# YtDownloader Assistant — Deployment Checklist

## 1. DeepSeek API key

Set via AWS SSM Parameter Store (or `.env` on the EC2 host):

```bash
# Never commit this value
DEEPSEEK_API_KEY=sk-...
```

The orchestrator returns **503** ("assistant is temporarily unavailable") when the key is unset, so the service boots safely without it.

Optionally override the model:
```bash
DEEPSEEK_MODEL=deepseek-chat   # default; set to deepseek-v4-flash if desired
```

## 2. Network egress

Ensure the EC2 security group allows **outbound TCP 443** to `api.deepseek.com`.  
The YtDownloader container calls `https://api.deepseek.com/v1/chat/completions` via `Microsoft.Extensions.AI`.

## 3. Database: `AssistantDailyUsage` table

YtDownloader does **NOT** run `Migrate()` at startup (only Identity and Scheduler do).  
The table must already exist in the `jiapp_ytdownloader` database.

**Migration on file:** `20260617192519_AddAssistantDailyUsage` is SQLite-typed.  
For **Postgres production**, the preferred drift-proof method is to generate Postgres-targeted SQL from EF:

```bash
dotnet ef migrations script \
    --project src/JiApp.YtDownloader \
    --context YtDbContext
```

If that is not available, apply the equivalent DDL manually — note the column and index names must match the EF entity exactly:

```sql
CREATE TABLE IF NOT EXISTS "AssistantDailyUsage" (
    "Id"            BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    "UserId"        BIGINT NOT NULL,
    "UsageDateUtc"  DATE NOT NULL,
    "Count"         INT NOT NULL DEFAULT 0
);

CREATE UNIQUE INDEX IF NOT EXISTS "IX_AssistantDailyUsage_UserId_UsageDateUtc"
    ON "AssistantDailyUsage" ("UserId", "UsageDateUtc");

CREATE INDEX IF NOT EXISTS "IX_AssistantDailyUsage_UserId"
    ON "AssistantDailyUsage" ("UserId");
```

> This table is queried by `IAssistantUsageRepository.TryConsumeAsync()` for daily quota enforcement (default: 30 messages/user/day).

**How to apply:** same method as the other YtDownloader tables. If you normally apply Yt schema changes via `psql -f`, include this DDL in the same script.

## 4. RAM safety

The concurrent assistant SSE stream cap is **1** (process-wide `SemaphoreSlim`).  
A second concurrent request receives **503** ("busy, try again") without consuming quota.

Optionally set a GC heap hard limit during load-testing (Phase F):
```bash
YT_GC_HEAP_LIMIT=0x8000000   # 128 MiB hex (example; tune to your instance)
```
Leave empty for no cap.

## 5. Deploy workflow

Reference the `deploy-ji-app-aws` skill/workflow for the full deploy sequence:

1. **Build & push** Docker images (`build-and-push.sh`)
2. **Copy** `docker-compose.prod.yml` + init scripts to the EC2
3. **Set** required env vars (`POSTGRES_PASSWORD`, `JWT_KEY`, `YOUTUBE_API_KEY`, `DEEPSEEK_API_KEY`)
4. **Apply** the `AssistantDailyUsage` table DDL
5. **Start** services: `docker compose -f docker-compose.prod.yml up -d`
6. **Verify** health: `GET /api/v1/yt/health`

## 6. Notes

- The assistant chat endpoint is at `POST /api/v1/yt/assistant/chat` (SSE, JWT-gated).
- The MCP tools endpoint is at `http://ytdownloader:6702/mcp` (internal-network only, NOT Gateway-routed).
- Both require the `module:YtDownloader` JWT claim.
